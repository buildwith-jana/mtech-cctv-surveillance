object classes can be added later without restructuring code (see "Extensibility" below).

## Architecture Decisions

1. **Model**: YOLO11 via `ultralytics==8.4.64` (latest stable at time of writing, verified
   to fully support YOLO11). Will not change model family without an explicit request, even
   as newer Ultralytics model generations (e.g. YOLO26) become available.
2. **Extensibility / no hardcoded classes**: No script hardcodes a class name or ID. All class
   identities are read at runtime from `configs/data.yaml` via `scripts/utils.py::load_class_names()`.
   Adding a class means editing `configs/data.yaml` and pointing to a dataset with that class —
   no script changes required. `scripts/verify_dataset.py` cross-checks label class IDs against
   `configs/data.yaml` and flags anything undeclared, which will catch schema mismatches the
   moment a second dataset (e.g. knives) is merged in.
3. **Notebook is an orchestrator only**: `notebooks/colab_train.ipynb` contains no logic beyond
   environment setup and `!python scripts/....py` calls. All actual logic (downloading,
   verifying, visualizing, training, evaluating, inference) lives in `scripts/*.py` so it is
   testable, versionable, and runs identically outside a notebook (e.g. on a laptop later).
4. **Git/GitHub from the start**: code is version-controlled in Git. Large binary artifacts
   (`data/`, `runs/`, generated `outputs/`) are gitignored and never committed — they're
   regenerated by scripts or, on Colab, persisted via a Google Drive symlink (see below).

## Running on Google Colab

1. Open `notebooks/colab_train.ipynb` in Colab (upload it directly, or open from GitHub via
   Colab's "GitHub" tab once the repo is pushed).
2. Runtime -> Change runtime type -> GPU.
3. Edit the `GITHUB_REPO_URL` variable in the notebook to your repo's URL.
4. Run cells top to bottom. The notebook clones your repo fresh each session (fast, always
   up to date) and symlinks `data/`, `runs/`, `outputs/` to a persistent folder in your Google
   Drive (`MyDrive/weapon-detection-storage`) so the dataset and trained weights survive
   across sessions without ever being committed to Git.
5. Get a free Roboflow API key at https://app.roboflow.com/settings/api when prompted.

## Extensibility: Adding a New Class Later (e.g. Knives)

1. Add the class to `configs/data.yaml`:
   ```yaml
   names:
     0: pistol
     1: knife
   nc: 2
   ```
2. Point `scripts/download_dataset.py` (or a new dataset-specific download call) at a
   knife-annotated dataset, using the same `--workspace/--project/--version` pattern.
3. Run `scripts/verify_dataset.py` — it will confirm both classes are recognized and flag
   any label using an undeclared class ID.
4. Re-run training (Phase 6) with `nc: 2` — no script code changes needed.

## Design Notes (for dissertation methodology section)

- **Why Roboflow Pistols for the first pass**: smaller, cleaner, single-class dataset lets
  Phases 1-5 (pipeline correctness) be validated fast before scaling to a larger, noisier
  multi-class dataset. OD-WeaponDetection or a merged dataset is a natural Module 2 extension.
- **Why single-class (pistol only) first**: isolates pipeline bugs from model-capacity/
  class-imbalance issues. Knife detection is deferred per your scope decision; the class-agnostic
  design above means that extension requires no architectural rework.
- **Why YOLO11 via Ultralytics**: single API surface for train/val/predict/export, actively
  maintained, and the exact same code path scales from this baseline to larger/multi-class
  datasets later without a rewrite.
